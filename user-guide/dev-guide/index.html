如何提交一个 mr

  1. 开发一个新特性之前，请先 建立 issue，写下技术方案, 原型等，跟小伙伴们讨论达成一致.

  2. 从 雨燕 获取最近的迭代分支，修改完毕后 mr 至该分支, 确保:

    1. 遵循已有编码风格和规范
    2. ci 通过 (附 使用 ava 进行测试的最佳实践)
    3. 包含对应的 js-sdk 和 文档 mr 链接

  3. 代码 review 通过后才可以合并, 请勿自行合并.

  4. mr 完成后关闭对应的 issue.

压测

benchmark 目录下存放了一些压测脚本，你可以基于这些已有的压测脚本来创建新的压测，并把脚本保存至此。
具体操作请参见 chair 的压测文档 。

发布策略

  1. 每周五下班前合并的所有 mr, 下周一或周二发布. 紧急 bugfix 单独发布.
  2. 不能在最近一个发布点内完成的 wip mr，从 master 检出，mr 至 master，待最终完成后 mr 至迭代分支.
  3. 发布前负责发布的同学需要提交一个 MR，整理这次发布的 changelog 到 History.md，并按照 semver 变更 package.json 中的版本. 记得同时更新 basement-doc 的 changelog.
  4. beta 发布后, 必须对所有功能进行验证, 然后 beta 2 小时以上, 观察无异常再全量发布.
